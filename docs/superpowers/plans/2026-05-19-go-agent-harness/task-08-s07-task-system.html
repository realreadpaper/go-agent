Task 8: 实现 s07 持久任务图

Prerequisites: Task 2 (agent loop)

Goal: 实现持久任务图，任务保存为 .tasks/task_N.json，支持 blockedBy 依赖、状态机驱动的状态转换和自动解锁。

给第一次接触 Agent 的读者

源教程 s07 从会话内 todo 升级到磁盘上的任务图。TodoWrite 适合“我现在这轮要做什么”，但真实项目会跨会话、跨 agent、跨目录运行。任务必须在上下文压缩和进程重启之后仍然存在。

新手先懂这几个概念

  • 任务图：任务之间可以有依赖关系，不是单纯列表。A 完成后，B 和 C 可能同时解锁。
  • blockedBy：一个任务被哪些前置任务阻塞。为空且状态为 pending 的任务就是 ready。
  • 状态机：任务状态不是随便写字符串，而是通过本地转换表校验。合法转换才会落盘，非法转换会返回清晰错误。
  • 磁盘持久化：每个任务一个 JSON 文件，方便恢复，也方便后续多个 agent 通过文件协调。

和源教程的对应关系

源 s07 把任务保存到 .tasks/ 并提供 create/update/list/get 工具。本任务用 internal/tasks/manager.go 实现同样的 CRUD、依赖解除和 ready 列表，给 s08 之后的并发与团队协作打基础。状态转换必须通过本地 Go 转换表完成，不引入外部状态机依赖，便于教学读者直接读懂规则。

完成后应该理解

你应该能区分 Todo 和 Task：Todo 是当前会话的短期计划，Task 是长期、可恢复、可分配的工作单元。后面的后台任务、团队和 worktree 都会围绕 Task 运转。你还应该能解释为什么这里需要状态机：长期任务会被多个 agent 或多次运行更新，只有显式转换表才能避免 completed 被误改回 pending 这类不可靠状态。

Context

本任务依赖 Task 2 的 tools.Registry。任务图是独立的持久化机制，不依赖 Todo 或 Subagent。

Steps

Step 1: 写任务图测试**

覆盖：

  • Create 生成递增 ID。
  • Update(id, completed) 自动从其他任务 blockedBy 移除该 ID。
  • ListReady() 只返回 pending、无 owner、blockedBy 为空的任务。
  • 状态仅允许 pending、in_progress、completed。
  • 状态转换必须走状态机：pending -> in_progress、pending -> completed、in_progress -> pending、in_progress -> completed 合法；completed 是终态，不能再转回其他状态。
  • 重复设置相同状态是幂等操作，不应触发错误或重复副作用。
Step 2: 实现任务模型** 
type Task struct {
	ID          int    `json:"id"`
	Subject     string `json:"subject"`
	Description string `json:"description"`
	Status      string `json:"status"`
	BlockedBy   []int  `json:"blockedBy"`
	Owner       string `json:"owner"`
	Worktree    string `json:"worktree"`
}

每个任务保存为 .tasks/task_.json。

状态机转换表本地存储在 Go 代码中，例如：

var allowedTransitions = map[string]map[string]bool{
	StatusPending: {
		StatusInProgress: true,
		StatusCompleted:  true,
	},
	StatusInProgress: {
		StatusPending:   true,
		StatusCompleted: true,
	},
	StatusCompleted: {},
}

实现要求：

  • Transition(current, next) 负责校验状态值和转换边。
  • current == next 视为幂等成功。
  • 非法转换必须返回包含当前状态和目标状态的错误。
  • 只有成功转换到 completed 后，才执行依赖解锁 action。
Step 3: 注册工具**

工具：

  • task_create(subject, description)
  • task_update(task_id, status, add_blocked_by, remove_blocked_by, owner, worktree)
  • task_get(task_id)
  • task_list()
Step 4: 实现 s07 CLI**

保留 s03 Todo，但 system prompt 说明：复杂、多轮、跨 agent 目标优先使用持久 task graph。

Run:

cd go-agent
go test ./internal/tasks
go run ./cmd/s07-task-system "Create 3 tasks: setup, write code, write tests. Make them depend on each other."

Expected: .tasks/task_1.json 等文件存在；依赖完成后自动解锁。

Step 5: 实现后真实验证任务**

无网络验证要覆盖任务创建、依赖阻塞、状态流转和磁盘恢复：

cd go-agent
go test -count=1 ./internal/tasks ./internal/tools
go vet ./internal/tasks

真实模型验证要创建一个带依赖的任务图，完成前置任务后再观察后续任务解锁：

cd go-agent
go run ./cmd/s07-task-system "Create three tasks: design API, implement API blocked by design, write tests blocked by implement. Mark design done and show which tasks are ready."

Expected: .tasks/ 中有独立 JSON 文件；被阻塞任务不会被列为 ready，依赖完成后状态可被重新计算。
