Task 5: 实现 s04 Subagent

Prerequisites: Task 2 (agent loop + registry)

Goal: 实现一次性子 agent runner，父 agent 通过 task 工具派发任务，只接收最终摘要。

给第一次接触 Agent 的读者

源教程 s04 引入 subagent，是为了解决上下文污染。父 agent 经常只想知道一个结论，比如“项目用什么测试框架”，但调查这个结论可能要读很多文件。让父 agent 自己读完所有文件，会把大量临时信息留在主上下文里。

新手先懂这几个概念

  • 父 agent：负责主任务和最终决策，拥有 task 工具。
  • 子 agent：用全新的 messages 启动，完成一个局部调查或操作，结束后丢弃自己的上下文。
  • 摘要返回：父 agent 只收到最终摘要，不收到子 agent 的全部工具调用历史。
  • 禁止递归 task：子 agent 不再拥有 task 工具，避免无限生成子任务。

和源教程的对应关系

源 s04 的 Python 版本用 run_subagent() 创建独立循环。本任务把它落到 internal/subagent/subagent.go：共享 LLM client 和基础工具，但隔离 messages，并用最大轮数保护执行边界。

完成后应该理解

你应该能判断什么时候该用 subagent：当任务需要大量探索，但父上下文只需要结论时。Subagent 的核心不是并行，而是上下文隔离和信息压缩。

Context

本任务依赖 Task 2 的 agent.Loop 和 tools.Registry。子 agent 复用相同的 loop 机制，但使用独立的 messages 和受限的工具集。

Steps

Step 1: 写 subagent 测试**

用 fake client 验证：

  • 子 agent 使用全新的 messages。
  • 父 agent 只收到最终摘要字符串。
  • 子 agent 工具集中没有 task，防止递归。
Step 2: 实现 Runner** 
type Runner struct {
	Client    llm.Client
	Model     string
	System    string
	Tools     *tools.Registry
	MaxRounds int
}

func (r *Runner) Run(prompt string) (string, error)

摘要提取逻辑：拼接最终 response 中所有 text block；为空返回 (no summary)。

Step 3: 注册父端 task 工具**

RegisterTask(reg, runner)，input schema:

{"type":"object","properties":{"prompt":{"type":"string"}},"required":["prompt"]}
Step 4: 实现 s04 CLI**

父端工具包含 base tools + task；子端工具仅包含 base tools。

Run:

cd go-agent
go test ./internal/subagent
go run ./cmd/s04-subagent "Use a subtask to find what testing framework this project uses"

Expected: 测试 PASS；父上下文只记录 task 的摘要结果。

Step 5: 提交** 
git add go-agent/internal/subagent go-agent/cmd/s04-subagent
git commit -m "feat(go): add subagent runner"
