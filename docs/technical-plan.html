1. 目标与边界

本项目不是把 12 篇文档简单翻译成 Go，而是实现一条真实可运行的 agent 演进链路。

目标

实现从单循环 agent 到任务图、后台执行、团队协作、自治认领和 worktree 隔离的完整链路。

非目标

不在首轮加入 GUI、不做分布式数据库、不追求多模型兼容层，不把所有能力先做成可插拔平台。

约束

测试必须可离线运行；所有网络依赖都要可替换；每个 commit 都要可回滚、可验证。

2. 架构分层

实现采用三层结构：执行循环层、工具与状态层、协作与持久化层。前两层支撑 s01-s06，第三层承接 s07-s12。

执行循环层

负责消息驱动、LLM 调用、tool_use 分发、subagent、压缩和提醒注入。

工具与状态层

负责 bash、文件读写、todo、skill、task、background 等能力的实现和沙箱控制。

协作与持久化层

负责任务图、队友生命周期、消息总线、请求-响应协议、自治认领和 worktree 绑定。

3. 文件与职责

以下文件边界会在实现里固定下来，避免把所有逻辑堆在一个大包里。

cmd/go-agent/main.go

CLI 入口、参数解析、配置加载和启动 agent。

internal/llm/

定义模型接口、Anthropic 实现、fake 实现和测试夹具。

internal/agent/

维护消息循环、上下文压缩、subagent、idle/work 状态切换。

internal/tools/

工具注册与分发、每个工具的输入输出契约、统一错误包装。

internal/state/

磁盘状态读写：todo、task、team、transcripts、worktrees。

docs/

总入口、详细方案和后续 12 篇章节文档，作为实现和审阅的可读输出。

4. 运行时流程

  1. CLI 读取配置，初始化工作目录、状态目录和 LLM client。
  2. 主循环将用户输入写入 messages，调用 LLM，解析返回内容。
  3. 如果出现 tool_use，按名称查找 handler，执行后把结果追加回 messages。
  4. 如果没有 tool_use，结束当前轮并返回文本输出。
  5. 在 s03 以后加入 todo reminder，在 s06 以后加入压缩，在 s08 以后加入后台通知 drain。
user -> agent loop -> llm -> tool dispatch -> tool_result -> llm -> ... -> final text

5. 测试策略

测试按能力分层，而不是只做一条大烟囱测试。关键行为都要有单元测试，跨模块行为在每个 commit 后补一条最小集成测试。

单元测试

工具解析、沙箱路径、状态转换、协议 FSM、压缩逻辑、任务图依赖解除。

集成测试

fake LLM 驱动 agent loop，验证 tool_result 回写、子 agent 摘要、后台通知和 inbox 注入。

文件系统测试

临时目录里验证 .tasks、.team、.worktrees、.transcripts 的真实落盘和恢复。

固定验证命令：go test ./... 和 go run ./cmd/go-agent --help。

7. 风险与决策点

LLM 依赖

生产环境使用的模型和 API 形状需要尽早定死，否则后面所有消息和工具协议都会漂移。

文件系统复杂度

task/team/worktree/transcript 四套状态都落盘，必须保持目录边界清晰，避免互相污染。

测试成本

背景线程、worktree 和协议 FSM 都要有可重复的集成测试，不然重构后很难判断回归点。

8. 验收标准

  • 12 个 commit 能一一对应 12 篇文档，且每个 commit 都能独立通过测试。
  • go test ./... 通过，go run ./cmd/go-agent --help 可运行。
  • 根目录 index.html、docs/index.html、docs/technical-plan.html 都可打开并互相可达。
  • 所有磁盘状态目录在空工作区下可初始化、可恢复、可清理。

如果这份方案通过审核，下一步就按 12 个 commit 顺序开始实现。
